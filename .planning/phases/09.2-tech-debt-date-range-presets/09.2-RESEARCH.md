## Architecture Patterns

### Pattern 1: API-to-Frontend Data Flow (9.2-01)

**What:** Expose server-side constants to the frontend via API response.

**Current state:** `PRICING_LAST_UPDATED` and `PRICING_SOURCE` are exported from `src/cost/pricing.ts` but only used in tests. The frontend (`web/src/`) is a separate Vite app that cannot import server-side modules directly.

**Approach:** Add the metadata fields to the existing `/api/v1/models` response (it already returns `{ rows, totalCost, unknownModels }`). Add two fields: `pricingLastUpdated: string` and `pricingSource: string`. This avoids creating a new endpoint for two static strings.

Alternative approach: Create a dedicated `/api/v1/pricing-meta` endpoint. This is cleaner separation but adds an extra network call. Given these are static values that never change during a server session, piggybacking on `/models` is acceptable.

**Recommended:** Augment `/api/v1/models` response. This is where pricing context is most relevant (models page shows cost breakdowns).

However, CostInfoTooltip is rendered on 3 pages (Overview, Models, Sessions). If we augment `/models`, the tooltip on Overview and Sessions pages would need to either: (a) also call `/models` just for metadata, or (b) use a separate lightweight endpoint.

**Best approach:** Create a tiny `/api/v1/pricing-meta` endpoint returning `{ lastUpdated: string, source: string }`. Create a `usePricingMeta()` hook with react-query caching (staleTime: Infinity since it never changes during a session). CostInfoTooltip calls this hook. This is cleanest because:
- CostInfoTooltip renders on 3 pages -- it should not depend on which page it's on
- The data is static for the server lifetime (no re-fetches needed)
- `staleTime: Infinity` means one fetch total

```typescript
// src/server/routes/api.ts -- new endpoint
import { PRICING_LAST_UPDATED, PRICING_SOURCE } from '../../cost/pricing.js';

app.get('/pricing-meta', (c) =>
  c.json({ lastUpdated: PRICING_LAST_UPDATED, source: PRICING_SOURCE })
);
```

```typescript
// web/src/hooks/usePricingMeta.ts
export function usePricingMeta() {
  return useQuery({
    queryKey: ['pricing-meta'],
    queryFn: async () => {
      const res = await fetch('/api/v1/pricing-meta');
      if (!res.ok) throw new Error('Failed to fetch pricing meta');
      return res.json() as Promise<{ lastUpdated: string; source: string }>;
    },
    staleTime: Number.POSITIVE_INFINITY, // never refetch -- static for server lifetime
  });
}
```

```typescript
// web/src/components/CostInfoTooltip.tsx
// Interpolate lastUpdated and source into tooltip text
export function CostInfoTooltip() {
  const { data } = usePricingMeta();
  // Tooltip text: "Based on API token pricing (last verified {date}, source). ..."
  // Graceful fallback if data not yet loaded: show existing hardcoded text
}
```

### Pattern 2: Prior Period Trend Computation (9.2-02)

**What:** Wire `usePriorSummary` into the Overview "Period Cost" StatCard to show percentage change vs prior equivalent period.

**Current state:** Overview.tsx lines 35-42 explicitly return `null` for `trendPercent` with a comment "Phase 5+ can add a usePriorSummary hook". The hook exists and is already wired into `CacheEfficiencyCard.tsx` (lines 13, 29-33). The pattern is proven and battle-tested.

**CacheEfficiencyCard reference pattern (lines 29-33):**
```typescript
const trendPercent: number | null = (() => {
  if (preset === 'all' || preset === 'custom') return null;
  if (currentScore === null || priorScore === null || priorScore === 0) return null;
  return ((currentScore - priorScore) / priorScore) * 100;
})();
```

**For Overview cost trend, the equivalent is:**
```typescript
const { data: priorSummary } = usePriorSummary(from, to);

const trendPercent: number | null = (() => {
  if (!periodSummary || !priorSummary) return null;
  if (preset === 'all' || preset === 'custom') return null;
  if (priorSummary.totalCost === 0) return null;
  return ((periodSummary.totalCost - priorSummary.totalCost) / priorSummary.totalCost) * 100;
})();
```

**Key considerations:**
- `usePriorSummary(from, to)` is disabled when from/to are undefined (the `all` preset case) -- `enabled: Boolean(from && to)` is built in
- Division by zero guard: `priorSummary.totalCost === 0` returns null (avoids Infinity)
- The `custom` preset should also return null (no meaningful "prior custom" period)
- TrendIndicator already handles null gracefully (renders "No prior data")
- **New 24h/48h presets from 9.2-03 will automatically work** because they set concrete from/to dates

### Anti-Patterns to Avoid
- **DO NOT import pricing.ts from frontend code:** The web/ directory is a separate Vite app with no path alias to src/. Always expose server data via API.
- **DO NOT add a separate API call per page for pricing metadata:** Use a single hook with staleTime: Infinity; react-query deduplication handles the rest.
- **DO NOT change the existing `presetToDates` function signature:** Only add branches to the if/switch; the return shape `{ from, to }` is stable.

## Common Pitfalls

### Pitfall 1: Division by Zero in Trend Calculation
**What goes wrong:** `priorSummary.totalCost` could be 0 (no usage in prior period), causing `Infinity` percentage.
**Why it happens:** New users with only one period of data.
**How to avoid:** Guard `if (priorSummary.totalCost === 0) return null;` before the division. TrendIndicator handles null gracefully.
**Warning signs:** NaN or Infinity showing in the UI.

### Pitfall 2: Preset Type Exhaustiveness
**What goes wrong:** Adding `'24h' | '48h'` to Preset but forgetting to handle them in `presetToDates()` or `getActiveLabel()`.
**Why it happens:** TypeScript union types don't always enforce exhaustive checks without explicit `never` assertions.
**How to avoid:** Add `default: { const _: never = preset; return { from: undefined, to: undefined }; }` to the exhaustive check, or restructure the if-chain. The `getActiveLabel` switch should also have a default.
**Warning signs:** Clicking 24h/48h does nothing or shows wrong label.

### Pitfall 3: Removing Events Route Breaks Tests
**What goes wrong:** Deleting the `/events` route might break test setup if any test fixture references `AppState.events`.
**Why it happens:** `AppState` has an `events: NormalizedEvent[]` field (used elsewhere in the server), and test fixtures set `events: []`.
**How to avoid:** Only delete the `app.get('/events', ...)` route handler in api.ts:335. Do NOT remove the `events` field from `AppState` -- it is used by other server code. The test fixtures set `events: []` because `AppState` requires it, not because they test the `/events` endpoint.
**Warning signs:** TypeScript compilation errors in test files.

### Pitfall 4: CostInfoTooltip Hook in Non-Component Context
**What goes wrong:** CostInfoTooltip uses a React hook (`usePricingMeta`) but is rendered inside `SortableTable Column.label` which was widened to `React.ReactNode` in Phase 9.1.
**Why it happens:** Since CostInfoTooltip is used as `<CostInfoTooltip />` (a React component), not as a plain string, this is fine -- React components can use hooks. The issue would only arise if CostInfoTooltip were used outside React render context.
**How to avoid:** Ensure CostInfoTooltip is always rendered as a React element (`<CostInfoTooltip />`), never called as a function.

### Pitfall 5: Hour-based Preset Date Calculation
**What goes wrong:** Using `setDate(getDate() - 1)` for 24h instead of millisecond arithmetic causes incorrect ranges when crossing month boundaries.
**Why it happens:** `setDate(getDate() - 1)` on March 1 gives February 28/29 correctly, but the time component is wrong -- it produces exactly midnight-to-midnight, not "24 hours ago from now".
**How to avoid:** Use millisecond arithmetic: `new Date(to.getTime() - 24 * 60 * 60 * 1000)`. This gives a precise 24-hour window ending at the current moment.

### Date Range Presets (9.2-03)

```typescript
// useDateRangeStore.ts
export type Preset = '24h' | '48h' | '7d' | '30d' | '90d' | 'all' | 'custom';

function presetToDates(preset: Preset): { from: Date | undefined; to: Date | undefined } {
  if (preset === 'all') return { from: undefined, to: undefined };
  const to = new Date();
  if (preset === '24h' || preset === '48h') {
    const hours = preset === '24h' ? 24 : 48;
    const from = new Date(to.getTime() - hours * 60 * 60 * 1000);
    return { from, to };
  }
  const days = preset === '7d' ? 7 : preset === '30d' ? 30 : 90;
  const from = new Date(to);
  from.setDate(from.getDate() - days);
  return { from, to };
}
```

## Open Questions

1. **CostInfoTooltip link behavior in CSP-restricted environment**
   - What we know: The server has CSP middleware blocking external requests. The tooltip source link (`<a href={source}>`) is a navigation link, not a fetch -- CSP `default-src` does not block anchor tag navigation.
   - What's unclear: Whether the CSP config includes `navigate-to` directive (unlikely, it's a deprecated proposal).
   - Recommendation: No action needed. Standard anchor links are not blocked by CSP. But the planner should verify the link opens correctly in the final build.

2. **Tooltip text length with metadata**
   - What we know: Current tooltip text is ~25 words. Adding "(last verified 2026-02-28)" adds ~4 words. Adding a "Source" link adds ~1 word.
   - What's unclear: Whether the w-64 (16rem) width is sufficient for the expanded text.
   - Recommendation: Keep w-64; the text wraps naturally with `text-left`. May need a slight width bump to w-72 if it looks cramped -- defer to implementation.
