Purpose: After Plan 01 created the provider infrastructure, this plan connects it to the existing server layer. The API routes, AppState, and public API all switch to consuming UnifiedEvent[]. This is the critical integration step that makes the abstraction real. After this plan, the server code compiles and all 68 server-layer tests pass against the new types.

Output: Updated `server.ts`, `api.ts`, `index.ts`, and 4 server test files, all consuming UnifiedEvent[].
</objective>

  <action>
    **Step 1: Update `src/server/server.ts` -- AppState interface.**

    Change AppState from:
    ```typescript
    export interface AppState {
      events: NormalizedEvent[];
      costs: CostEvent[];
      rawEvents?: NormalizedEvent[];
      showMessages?: boolean;
    }
    ```
    To:
    ```typescript
    export interface AppState {
      events: UnifiedEvent[];
      providers: ProviderInfo[];
      showMessages?: boolean;
    }
    ```

    Update imports: remove NormalizedEvent, CostEvent. Add import of `UnifiedEvent` and `ProviderInfo` from `'../providers/types.js'`.

    The `createApp` function signature stays the same (takes AppState, returns Hono). The function body is unchanged (it just passes state to apiRoutes).

    **Step 2: Rewrite `src/server/routes/api.ts` -- the big one (1080 lines).**

    This is a systematic find-and-replace with type adjustments. Key changes:

    a) **Imports**: Remove `import type { CostEvent } from '../../cost/types.js'` and `import type { NormalizedEvent } from '../../parser/types.js'`. Add `import type { UnifiedEvent } from '../../providers/types.js'`. Keep `import { PRICING_LAST_UPDATED, PRICING_SOURCE } from '../../providers/claude/cost/pricing.js'` (update path from `../../cost/pricing.js`).

    b) **Every reference to `state.costs`** becomes `state.events`. There are approximately 20 occurrences. The variable names in route handlers should also change for clarity:
    - `let costs = state.costs` -> `let events = state.events`
    - `costs.filter(...)` -> `events.filter(...)`
    - `costs.reduce(...)` -> `events.reduce(...)`
    - But be careful: some local variables are named `costs` for clarity. Rename them to `events` throughout for consistency.

    c) **`state.rawEvents` elimination**: The chat endpoints currently check `state.rawEvents`. After this change:
    - `/chats` and `/chats/:id` check `state.showMessages` to gate access (same 403 logic)
    - Instead of filtering `state.rawEvents`, filter `state.events` and check for `e.message !== undefined` to find events with conversation content
    - Remove all references to `state.rawEvents`

    d) **Type annotations**: Change all `NormalizedEvent` type annotations in the file to `UnifiedEvent`. This affects:
    - `extractFirstUserMessage(events: NormalizedEvent[])` -> `extractFirstUserMessage(events: UnifiedEvent[])`
    - `getTextContent(events: NormalizedEvent[])` -> `getTextContent(events: UnifiedEvent[])`
    - `sessionMap` types
    - Chat endpoint internal types

    e) **`e.costUsd`** is now a plain `number` on UnifiedEvent (was `EstimatedCost` branded type). No code change needed in the arithmetic -- the branded type was phantom and arithmetic worked the same way.

    f) **`e.unknownModel`** field: On UnifiedEvent this is `boolean | undefined` (was `true | undefined` on CostEvent). The existing check `e.unknownModel && e.model` still works. But the type might need adjustment. Check if `unknownModel?: boolean` on UnifiedEvent satisfies the existing `if (e.unknownModel && ...)` pattern -- it does (truthy check).

    g) **Chat endpoints**: The chat detail endpoint constructs a summary that now needs a `provider` field per CONTEXT.md ("Chat responses include provider field in summary"). Add `provider: sorted[0]?.provider ?? 'claude'` to the summary object. The chat list endpoint can also include provider in each ChatItem.

    h) **`assignProjectNames` and helper functions**: These accept `NormalizedEvent[]` internally. Change parameter types to `UnifiedEvent[]`. The `uuid` field is now `id` on UnifiedEvent -- but these helpers don't use uuid/id directly, so no functional change.

    IMPORTANT: Do NOT change any computation logic (date bucketing, session grouping, cost aggregation algorithms). Only types and field references change.

    **Step 3: Rewrite `src/index.ts` -- new public API.**

    Replace the entire file. The old API exported parseAll, computeCosts, applyPrivacyFilter, NormalizedEvent, CostEvent, etc.

    New public API per CONTEXT.md locked decision:
    ```typescript
    // Provider types
    export type { UnifiedEvent, ProviderId, CostSource, ProviderAdapter, ProviderInfo, LoadOptions } from './providers/types.js';

    // Main entry point
    export { loadProviders } from './providers/registry.js';
    ```

    Remove all old exports (parseAll, computeCosts, applyPrivacyFilter, NormalizedEvent, CostEvent, EstimatedCost, KnownModel, ParseOptions). These are now internal to the Claude adapter.

    The `enableDebug` import and `parseAll` function body are removed -- that logic now lives in the Claude adapter.

    ```typescript
    import type { UnifiedEvent } from '../../providers/types.js';

    function makeUnifiedEvent(
      costUsd: number,
      overrides?: Partial<UnifiedEvent>,
    ): UnifiedEvent {
      return {
        id: `test-uuid-${Math.random()}`,
        provider: 'claude',
        sessionId: 'session-1',
        timestamp: '2024-01-01T00:00:00Z',
        type: 'assistant',
        costUsd,
        costSource: 'estimated',
        ...overrides,
      };
    }
    ```

    Remove imports of `CostEvent`, `toEstimatedCost`, `NormalizedEvent` from old paths.

    **Step 2: Update AppState construction in all test files.**

    Old pattern: `{ events: [], costs: [] }` or `{ events: [], costs: [makeCostEvent(...)], rawEvents: [...], showMessages: true }`

    New pattern: `{ events: [], providers: [] }` or `{ events: [makeUnifiedEvent(...)], providers: [], showMessages: true }`

    The `providers` field should be `[]` in most tests (API routes don't use it). For completeness, provide a minimal array: `[{ id: 'claude', name: 'Claude Code', status: 'loaded', eventCount: N }]`.

    **Step 3: Update `api.test.ts` (22 tests).**

    - Replace all `makeCostEvent(...)` calls with `makeUnifiedEvent(...)`. The arguments are the same (costUsd, timestamp).
    - Where `makeCostEvent` accepted separate `tokens` param, pass via overrides: `makeUnifiedEvent(0.5, { tokens: {...}, timestamp: '...' })`
    - Remove unused `makeEvent` function (it's merged into makeUnifiedEvent)
    - Update `SummaryBody`, `CostOverTimeBody` type aliases if needed (they describe API response shapes, not internal types -- likely unchanged)

    **Step 4: Update `api-sessions.test.ts` (24 tests).**

    - Same helper replacement pattern
    - Session tests heavily use `sessionId`, `model`, `isSidechain`, `durationMs`, `cwd`, `gitBranch` -- all fields that exist on UnifiedEvent with same names
    - The `uuid` field was on NormalizedEvent; now it's `id` on UnifiedEvent. Test data that sets `uuid` should set `id` instead.

    **Step 5: Update `api-chats.test.ts` (15 tests).**

    - Chat tests currently construct `rawEvents` with message content. Now message content lives on the UnifiedEvent `message` field.
    - Old: `state = { events: filtered, costs: [...], rawEvents: eventsWithContent, showMessages: true }`
    - New: `state = { events: eventsWithContent, providers: [], showMessages: true }` where each event has both `costUsd`/`costSource` AND `message` field.
    - The `makeUnifiedEvent` helper needs to support `message` field in overrides.
    - Chat search tests check text content -- ensure `message` field is set on test events.

    **Step 6: Update `server.test.ts` (7 tests).**

    - Simple: change `{ events: [], costs: [] }` to `{ events: [], providers: [] }` in all AppState constructions
    - These tests just check that createApp returns a Hono instance and routes return correct status codes
