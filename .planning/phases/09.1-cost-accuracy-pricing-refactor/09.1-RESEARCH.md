# Phase 9.1: Cost Accuracy & Pricing Refactor - Research

**Researched:** 2026-03-04
**Domain:** UI labelling, tooltip patterns, pricing config architecture, API response augmentation
**Confidence:** HIGH

## Summary

This phase is a focused refactor with three distinct deliverables: (1) info tooltips on cost-related section headings to explain the pricing basis, (2) an unknown model warning on the Models page, and (3) a structural refactor of `pricing.ts` into a config-layer + lookup-layer architecture. All three are low-risk, additive changes with no new dependencies required.

The codebase already has all the building blocks: CSS-only group-hover tooltips used in at least 3 places (`Models.tsx:86-104`, `Sessions.tsx:28-30`, `CostBarChart.tsx:87-89`), the `CostEvent.unknownModel?: true` field already set by `engine.ts:36`, and a well-structured `pricing.ts` that just needs reorganization. The main challenge is surgical precision -- touching many files (6+ pages render `est.`) but changing each one minimally, and ensuring the API models endpoint surfaces unknown model data without breaking existing consumers.

**Primary recommendation:** Execute in three sequential waves: (1) pricing.ts refactor (backend-only, no UI change), (2) API augmentation for unknown models + Models page warning, (3) info tooltips on section headings. This ordering ensures each wave is independently testable and the most structurally risky change (pricing refactor) lands first.

### Discretion Recommendation: Grouped Tier Objects (Single File)

**Decision:** Use grouped tier objects in a single `pricing.ts` file with two clearly separated sections. Rationale:

1. **Grouped tiers** are more readable than individual constants -- ~7 named tier objects vs 30+ individual constants:
   ```ts
   const TIER_OPUS: ModelPricing = { inputPerMTok: 5.0, outputPerMTok: 25.0, ... };
   ```
2. **Single file** avoids import chain changes -- `engine.ts` already imports from `./pricing.js` and this import path stays stable.
3. Two files would require either re-exports or import changes in `engine.ts`, adding unnecessary diff surface.

### Discretion Recommendation: Info icon placement AFTER label text

**Decision:** Place the info icon after the label text (`All-time est. [i]`) rather than before. Rationale: the label text should be the primary visual element; the icon is supplementary. This matches the `cursor-help` underline pattern used in Models.tsx where the interactive element follows the content.

### Discretion Recommendation: Do NOT add PRICING_LAST_UPDATED to tooltip text

**Decision:** Do not include the pricing last-updated date or source URL in the tooltip. The tooltip text is already long. Adding "Prices last verified 2026-02-28" would make it even longer. The user explicitly deferred "Showing PRICING_LAST_UPDATED somewhere in the UI" to a future phase. Keep the tooltip focused on the Pro/Max explanation.

### Anti-Patterns to Avoid
- **Changing "est." format:** CONTEXT.md explicitly locks the `$X.XX est.` format. Do not change it to "API equiv." or any other suffix.
- **Tooltip on every cost value:** Only on section headings -- not on each `$X.XX est.` inline.
- **Yellow warning banner:** User specified subtle inline text, not a banner component.
- **Breaking MODEL_PRICING export:** `engine.ts` imports `MODEL_PRICING` -- the refactored version must export the same name with the same type.
- **Two-file split:** Adding a separate `pricing-config.ts` file creates unnecessary import chain changes. Keep one file.
- **SortableTable emptyMessage using ReactNode label:** The `emptyMessage` prop and column key handling are unrelated to `label` type widening. Do not conflate them.

## Common Pitfalls

### Pitfall 1: Breaking engine.ts imports
**What goes wrong:** Refactoring pricing.ts changes the export name or shape, breaking `engine.ts` which imports `MODEL_PRICING` and `ModelPricing`.
**Why it happens:** Easy to rename exports during refactoring.
**How to avoid:** Keep `MODEL_PRICING`, `ModelPricing`, and `KnownModel` as named exports from `pricing.ts`. The new `PRICING_LAST_UPDATED` and `PRICING_SOURCE` are additive exports only.
**Warning signs:** TypeScript compilation errors in `engine.ts`, test failures in `engine.test.ts`.

### Pitfall 2: Tooltip text overflow on narrow viewports
**What goes wrong:** The tooltip text ("Based on API token pricing. If you're on Claude Pro or Max...") is long and may overflow or get cut off.
**Why it happens:** CSS-only tooltips use `whitespace-nowrap` in the existing pattern.
**How to avoid:** Use fixed-width `w-64` with default wrapping instead of `whitespace-nowrap` for this specific tooltip. The token breakdown tooltip in Models.tsx uses `whitespace-nowrap` but that text is short. The CostInfoTooltip component should use `w-64 text-left` (256px width with normal text wrapping).
**Warning signs:** Tooltip extends past viewport edge on narrow screens.

### Pitfall 3: unknownModels response field breaking existing consumers
**What goes wrong:** Adding a new field to the `/api/v1/models` response could theoretically break strict type checking in the frontend if the `ModelsData` type doesn't expect it.
**Why it happens:** TypeScript strict mode + explicit type annotations.
**How to avoid:** Add `unknownModels` as an optional field to the `ModelsData` interface in `useModels.ts`: `unknownModels?: { models: string[]; sessionCount: number } | null`. The `useQuery<ModelsData>` generic ensures the field is type-checked.
**Warning signs:** TypeScript errors in useModels.ts or Models.tsx.

### Pitfall 4: Tier reference identity in tests
**What goes wrong:** Tests that compare `MODEL_PRICING['claude-opus-4-6']` with `===` to another entry might break if they now reference the same tier object.
**Why it happens:** Before refactor, each entry was a separate object literal. After refactor, aliases reference the same tier constant.
**How to avoid:** Existing tests in `pricing.test.ts` use `.toBe()` on individual numeric properties (e.g., `expect(p?.inputPerMTok).toBe(5.0)`) which continue to work. No tests use object identity comparison. The `satisfies` pattern preserves the record shape.
**Warning signs:** None expected, but run full test suite after refactoring.

### Pitfall 5: Unknown model warning not respecting date filter
**What goes wrong:** The unknown model warning shows all-time unknown models instead of only those in the selected date range.
**Why it happens:** If unknownModel aggregation is done on `state.costs` instead of the already-filtered `costs` variable.
**How to avoid:** Use the same `costs` variable that's already been filtered by `from`/`to` in the `/models` route (lines 219-221 of `api.ts`). The aggregation loop must come AFTER the date filtering.
**Warning signs:** Warning shows models that shouldn't appear for the selected date range.

### Pitfall 6: Info icon inconsistency across dark/light mode
**What goes wrong:** The info icon color doesn't match the surrounding text in dark mode.
**Why it happens:** Using hardcoded colors instead of the established dark mode token variables.
**How to avoid:** Use `text-slate-400 dark:text-[#6e7681]` for the SVG icon (slightly dimmer than label text `dark:text-[#8b949e]`, indicating it's supplementary). Use `fill="currentColor"` on the SVG so it inherits the text color.
**Warning signs:** Icon appears invisible or too bright in dark mode.

### Pitfall 7: Column label type widening breaking SortableTable
**What goes wrong:** Changing `Column.label` from `string` to `React.ReactNode` could break places that depend on `label` being a string (e.g., `emptyMessage` handling or key generation).
**Why it happens:** Implicit string assumptions in JSX.
**How to avoid:** Verified: `SortableTable.tsx:79` renders `{col.label}` in JSX which accepts ReactNode. The column React key at line 71 uses `String(col.key)-${colIdx}`, not the label. No `typeof col.label === 'string'` checks exist. The widening is safe.
**Warning signs:** TypeScript errors at call sites of SortableTable where `label: string` is passed (these should still satisfy `React.ReactNode`).

## Supplementary Research: Subscription Pricing & Competitor Analysis

**Researched:** 2026-03-04 (targeted online research)

### Subscription Per-Token Pricing: Does It Exist?

**No.** Anthropic publishes per-token rates **only for the API**. Pro ($20/mo) and Max ($100-$200/mo) are flat-fee subscriptions with usage limits -- no per-token billing.

Key official findings:
- Claude Code docs explicitly state: `/cost` data "isn't relevant for billing purposes" for Pro/Max subscribers ([Claude Code Cost Management](https://code.claude.com/docs/en/costs))
- Help center describes limits only in **relative multipliers** (5x, 20x vs Pro), never absolute token counts ([Usage Limits](https://support.claude.com/en/articles/11647753-understanding-usage-and-length-limits))
- Pro plan article says "at least five times the usage per session compared to free" -- no absolute numbers ([Pro Plan](https://support.claude.com/en/articles/8325606-what-is-the-pro-plan))
- Max plan article says "5 times more usage per session than Pro" (5x) or "20 times more" (20x) -- relative only ([Max Plan](https://support.claude.com/en/articles/11049741-what-is-the-max-plan))

**One exception: "Extra Usage" overage.** When Pro/Max subscribers exhaust their allocation, they can opt into consumption-based billing at **standard API rates** ([Extra Usage](https://support.claude.com/en/articles/12429409-extra-usage-for-paid-claude-plans)). This means API prices are technically accurate for overage scenarios.

### Can We Derive Effective Per-Token Pricing from Quotas?

**No -- structurally unsound.** The approach fails because:

1. **The denominator is unknowable.** Anthropic deliberately does not publish token quotas. Limits use rolling 5-hour windows + weekly caps (added August 2025), not a fixed monthly budget.
2. **Community estimates are rough:** ~44K tokens/5hr window (Pro), ~88K (Max 5x), ~220K (Max 20x) -- but these are reverse-engineered observations, not specs.
3. **Limits are not stable.** Anthropic changes them regularly (weekly caps added Aug 2025, holiday bonus episode Dec 2025, ongoing adjustments). Any derived rate becomes stale immediately.
4. **Subscriptions price access, not tokens.** The fee includes UI, conversation history, Projects, artifacts, priority access. Reducing to per-token strips non-token value.
5. **It incentivizes wrong optimization.** Per-token thinking encourages maximizing throughput -- the exact behavior that led Anthropic to add weekly caps.

**The honest framing** (per FinOps Foundation and credible analyses): "API-equivalent value received" -- not "effective per-token cost." Show what the usage *would have cost* at API rates alongside the subscription fee, as an ROI indicator.

Sources:
- [FinOps Foundation: How Token Pricing Really Works](https://www.finops.org/wg/genai-finops-how-token-pricing-really-works/)
- [Portkey: Claude Code Limits](https://portkey.ai/blog/claude-code-limits/)
- [The Register: Claude devs complain about surprise usage limits](https://www.theregister.com/2026/01/05/claude_devs_usage_limits/)

### How Competitors Handle Subscription vs API Pricing

Surveyed **15 open-source Claude Code usage/cost tracking tools**. Summary:

| Approach | Tools | Stars |
|---|---|---|
| **Ignore entirely** -- show API prices, no disclaimer | ccusage, ccost, ccboard, Claud-ometer, claude-code-otel, ClaudeUsageTracker | 11.2K+ |
| **Track limits only** -- avoid dollar amounts entirely | claude-code-limit-tracker, Claude-Code-Usage-Monitor | 6.8K+ |
| **"Hypothetical API costs" label** | claude-code-stats | 2 |
| **"Was it worth it?" ROI framing** | cctrack (blog post) | 2 |
| **Detect plan, don't adjust costs** | CCSeva (macOS menu bar) | 780 |
| **Skip costs entirely** | Sniffly (Chip Huyen) | 1.2K |

**Nobody solves this well.** The dominant tool (ccusage, 11.2K stars) ignores subscription pricing entirely. Key competitors:

- **ccusage** (11.2K stars): Uses LiteLLM pricing database. Three modes: `auto` (uses `costUSD` from JSONL when available, else calculates), `calculate`, `display`. Zero subscription awareness. No disclaimer.
- **Claude-Code-Usage-Monitor** (6.8K stars): Tracks subscription tier rate limits (Pro/Max5/Max20) but decouples this from cost calculation entirely.
- **cccost** (22 stars): Explicitly acknowledges the gap -- exists because "Claude Code does not show cost for Pro/Max users via `/cost`" -- but still shows API-equivalent costs without disclaimer.
- **claude-code-stats** (2 stars): **Most thoughtful approach found.** Labels costs as "hypothetical API costs" and lets users configure a `plan_history` array for comparison/ROI analysis.
- **cctrack** (2 stars): Author wrote a blog post framing it as "was my Max subscription worth it?" -- shows $1,428 API-equivalent vs $200 subscription cost.

### Implications for Phase 9.1

1. **Our planned approach (info tooltip with "API token pricing" disclaimer) is the right one.** It's more honest than what 90%+ of competitors do (showing API costs with no disclaimer at all).
2. **Do NOT derive subscription per-token rates.** The numbers would be structurally misleading and immediately stale.
3. **This is a differentiation opportunity.** No major competitor has a proper disclaimer. Even a small tooltip puts us ahead of ccusage (11.2K stars).
4. **Future phase opportunity:** A "was my subscription worth it?" ROI view (show API-equivalent spend vs subscription cost) would be the best-in-class approach -- but that requires user plan configuration, which is correctly deferred.
5. **The "Extra Usage" exception** is worth noting: for subscribers who opt into overage billing, our API-based estimates are actually accurate. The tooltip text should not overstate the inaccuracy.
