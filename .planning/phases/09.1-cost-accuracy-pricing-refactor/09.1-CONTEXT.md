# Phase 9.1: Cost Accuracy & Pricing Refactor - Context

**Gathered:** 2026-03-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix how cost estimates are communicated to Pro/Max subscribers, and refactor `pricing.ts` into a maintainable config structure. This phase does NOT add subscription detection or a settings UI — that is deferred. It delivers: honest cost labelling with info tooltips, visible unknown-model warnings, and a split pricing config layer.

</domain>

<decisions>
## Implementation Decisions

### Cost labelling
- Keep the `$X.XX est.` suffix as-is everywhere — no copy change to the format
- Add a small ⓘ info icon next to **section headings only** — not every individual cost value
- Placement: StatCard labels ("All-time est.", "Selected period est."), the Models page total, and the Sessions "Cost" column header
- Tooltip text explicitly mentions Pro and Max by name: "Based on API token pricing. If you're on Claude Pro or Max, your actual spend is your subscription fee — not these numbers."
- Tooltip uses the existing CSS-only group-hover pattern (same as token breakdown tooltip in Models.tsx) — no tooltip library

### Unknown model warning
- Surface a warning on the **Models page only** — not Overview or inline per-session
- Visual style: **subtle inline text** below the Models table (not a yellow banner) — matches low-noise philosophy
- Show the actual unknown model IDs: "⚠ 2 sessions used unrecognized models (claude-5-opus, claude-4-ultra) — costs may be understated"
- Warning **respects the global date range filter** — consistent with how Models page data already works
- The API already returns `unknownModel` on cost events; the UI needs to consume and aggregate it

### Pricing config structure
- `lastUpdated`: exported constant — `export const PRICING_LAST_UPDATED = '2026-02-28';` (machine-readable, can be shown in UI or used by scripts)
- Pricing source URL: exported constant — `export const PRICING_SOURCE = 'https://...';`
- Keep the `ModelPricing` interface and `KnownModel` type in the same module
- No automated CI price drift check in this phase (deferred)

### Claude's Discretion
- Tier organization approach (grouped tier objects vs individual price constants)
- File split (single file with two sections vs. two separate files)
- Exact ⓘ icon placement within the layout (before or after the label text)
- Whether to add `PRICING_LAST_UPDATED` or `PRICING_SOURCE` to the tooltip text

</decisions>

<specifics>
## Specific Ideas

- Roadmap explicitly says: "honest relabelling is a low-risk improvement" — keep the change minimal
- The `$X.XX est.` format is already consistent everywhere; don't break that consistency
- Tooltip text should mention Pro and Max by name so users recognize their plan
- The unknown model warning listing IDs helps users understand which models need pricing updates
- Unknown model warning should list the distinct model IDs, not repeat per-session

</specifics>

## Existing Code Insights

### Reusable Assets
- `src/cost/types.ts:31` — `CostEvent.unknownModel?: true` already exists; needs UI consumption
- `web/src/components/StatCard.tsx` — accepts `label` (string) and `value` (string); ⓘ tooltip could go into a wrapper around the label text

### Established Patterns
- Cost values pre-formatted as `"$X.XX est."` strings before passing to components — tooltip goes on the label, not the value
- `"est."` appears in ~15 places across 6 files — keep consistent, only add ⓘ at section heading level
- CSS-only tooltips use: `relative group cursor-help` parent with `absolute hidden group-hover:block` child
- All aggregation is server-side — `unknownModel` data needs to flow through `/api/v1/models` response

### Integration Points
- `src/cost/engine.ts:34-37` — already sets `unknownModel: true` on events with unrecognized models
- `src/cost/pricing.ts` — the only file being structurally changed; `engine.ts` imports from it and must remain compatible
- Models API route — needs to surface unknown model count and IDs in its response
- `web/src/hooks/useModels.ts` — hook needs to expose unknown model data to the Models page
